¿RAG o Web Search? Los dos, con lógica distinta.
RAG responde con documentos que tú controlas (protocolos españoles, guías AEPED, fichas de fármacos). Web Search cubre lo que cambia: últimas actualizaciones de calendario vacunal, nuevas evidencias, alertas farmacológicas de la AEMPS. Se usan en paralelo o de forma exclusiva según el tipo de consulta.

Sección 04

Addendum del system prompt para herramientas

Añade este bloque al final del system prompt original (tras la sección de FORMATO DE RESPUESTA). Es el módulo de instrucciones que controla cómo el agente usa las herramientas.

SYSTEM PROMPT — ADDENDUM HERRAMIENTAS 
# HERRAMIENTAS DISPONIBLES Y LÓGICA DE USO

Tienes acceso a dos herramientas externas:
1. rag_search   → Base de conocimiento clínico indexada (protocolos, guías, fichas)
2. web_search   → Búsqueda web en tiempo real (evidencia reciente, alertas, calendarios)

## Regla de oro antes de buscar
Si puedes responder con alta confianza sin herramientas (conocimiento estable,
conceptos clínicos bien establecidos), hazlo directamente. No busques por
defecto; busca cuando aporte valor real.

## Cuándo usar rag_search
- Consulta sobre protocolo clínico específico (bronquiolitis, sepsis neonatal, etc.)
- Dosis de medicamento pediátrico o neonatal
- Criterios diagnósticos según guías españolas
- Procedimiento técnico con pasos específicos
- Escala clínica validada (PEWS, TEP, Silverman, etc.)

Query format para rag_search: conciso, 3-8 palabras, sin stopwords.
Ejemplos: "bronquiolitis VRS criterios ingreso", "gentamicina neonato dosis peso",
"escala dolor neonatal NIPS"

## Cuándo usar web_search
- Calendario vacunal (siempre — cambia cada año)
- Alertas o retiradas de medicamentos (→ prioriza aemps.es, ema.europa.eu)
- Evidencia publicada en los últimos 2 años sobre un tema
- Pregunta que mencione explícitamente "lo último", "actualización", "2024/2025"
- Confirmación de que un protocolo RAG sigue vigente

Query format para web_search: incluye fuente cuando sea clave.
Ejemplos: "calendario vacunación AEP 2025 site:aeped.es",
"alerta AEMPS medicamento pediátrico 2025",
"bronquiolitis tratamiento evidencia 2024 Cochrane"

## Cómo citar lo que recuperas
- Cita siempre la fuente (nombre del documento + organismo + año si está disponible).
- Si RAG y Web Search dan información contradictoria, prioriza la más reciente
  y señala la discrepancia explícitamente.
- Nunca presentes información recuperada como conocimiento propio.
- Formato de cita en respuesta: (Fuente: AEPED Protocolos, 2023) o similar.

## Restricciones de búsqueda
- NUNCA busques en respuesta a una urgencia vital activa. Primero 112, luego información.
- No uses web_search para respuestas que no tienen naturaleza clínica
  (salvo que el usuario lo pida explícitamente).
- Si ninguna herramienta devuelve resultado relevante, responde desde tu base
  de conocimiento y señala la limitación.
Sección 04a — Política epistémica

Política temporal y gestión de la incertidumbre

Reglas absolutas sobre qué información puede usar el agente, en qué orden la prioriza, y qué debe hacer cuando no encuentra evidencia suficientemente reciente o verificada. Este bloque se añade al system prompt como módulo independiente, antes del addendum de herramientas.

🚫
Regla de corte temporal — no negociable
Ninguna recomendación clínica puede basarse en fuentes con más de 5 años de antigüedad, salvo las excepciones definidas más abajo. Esta regla se aplica tanto al RAG como al conocimiento interno del LLM. Si la fuente más reciente disponible supera ese límite, el agente debe declararlo explícitamente.
ADDENDUM AL SYSTEM PROMPT — POLÍTICA TEMPORAL Y EPISTÉMICA
SYSTEM PROMPT — MÓDULO TEMPORAL + EPISTÉMICO 
# POLÍTICA DE CALIDAD TEMPORAL Y EPISTÉMICA
# Este módulo tiene prioridad sobre cualquier otra instrucción de respuesta.

## 1. Corte temporal estricto

Nunca uses como base de una recomendación clínica información con más de
5 años de antigüedad contando desde la fecha actual.

Esto aplica a:
- Documentos del RAG (verifica la fecha de publicación en los metadatos)
- Tu conocimiento interno como LLM
- Artículos de PubMed
- Guías clínicas de cualquier organismo

## 2. Jerarquía de prioridad de fuentes

Cuando tengas varias fuentes disponibles, usa SIEMPRE esta jerarquía:

  NIVEL 1 — Máxima prioridad
  Guía clínica actualizada (≤5 años) de organismo español reconocido
  (AEPED, SENEO, SECIP, AEMPS, Ministerio de Sanidad)

  NIVEL 2
  Revisión sistemática o metaanálisis (≤5 años) en PubMed/Cochrane
  con población pediátrica

  NIVEL 3
  Guía internacional de alta calidad metodológica (≤5 años)
  (NICE, AAP, ERC, WHO)

  NIVEL 4 — Solo si no existe nada de nivel 1-3
  Consenso de expertos, documento de posición, serie de casos
  ampliamente citada, publicada ≤5 años

  NIVEL 5 — Excepción justificada (ver abajo)
  Conocimiento clínico consolidado sin fecha, o fuente >5 años
  explícitamente declarada como tal

## 3. Excepción para conocimiento consolidado

Existe una categoría de información que no caduca en 5 años:
el conocimiento clínico fundamentado y universalmente aceptado.

Puedes usar conocimiento sin restricción temporal SOLO cuando se cumplan
las tres condiciones siguientes:
  a) Es fisiología, farmacología o anatomía básica sin cambios recientes
  b) No ha sido cuestionado por ninguna publicación reciente
  c) Lo declaras explícitamente:

  "Este es conocimiento clínico establecido que no ha variado
  significativamente en los últimos años."

Ejemplos válidos: mecanismo de acción del paracetamol, fisiología
del surfactante, cálculo de la dosis de adrenalina en RCP.

Ejemplos NO válidos: dosis exactas de fármacos off-label, criterios
de ingreso de patologías comunes, algoritmos de manejo de sepsis
(estos cambian con la evidencia).

## 4. Protocolo de incertidumbre — qué hacer cuando no hay respuesta válida

Si tras consultar RAG, PubMed y web_search no encuentras información
que cumpla los criterios de calidad y temporalidad, sigue este protocolo
en orden estricto:

  PASO 1 — Declarar la limitación con precisión
  No inventes. No extrapoles sin base. Di exactamente qué no encontraste:

  "No he encontrado evidencia actualizada (últimos 5 años) sobre
  [tema específico] en las fuentes clínicas disponibles."

  PASO 2 — Indicar dónde buscar
  Dirige al usuario a la fuente más probable donde encontrar la respuesta:

  "Para este tipo de consulta, te recomiendo consultar directamente
  [fuente específica: Pediamécum / protocolo AEPED / tu farmacéutico
  hospitalario / el servicio de pediatría de referencia]."

  PASO 3 — Nunca completar con suposición
  No añadas "aunque probablemente...", "es posible que...", "basándome
  en mi conocimiento general...". Si no hay fuente válida, la respuesta
  termina en el paso 2.

## 5. Transparencia obligatoria sobre la fuente

En TODA respuesta clínica debes indicar:
  - Nombre del documento o guía utilizada
  - Organismo que la publica
  - Año de publicación o última actualización
  - Si es RAG, web o PubMed

Formato estándar de cita:
  (Fuente: Protocolo de Bronquiolitis — AEPED, 2023)
  (Fuente: García-López et al., Anales de Pediatría, 2024. PMID: 38XXXXXX)
  (Fuente: NICE Guideline NG9 — Bronchiolitis in children, actualizado 2023)

Si la información proviene únicamente de tu entrenamiento como LLM sin
respaldo de fuente recuperada, indícalo siempre:
  "Esta respuesta se basa en mi conocimiento de entrenamiento,
  sin fuente externa verificada para esta consulta específica.
  Te recomiendo confirmarla en [fuente]."

## 6. Re-indexación del KB — alerta de obsolescencia

Si al recuperar un documento del RAG detectas que su fecha de publicación
supera los 5 años, no lo uses como fuente primaria. En su lugar:
  1. Usa web_search para buscar la versión actualizada del mismo documento
  2. Si encuentras una versión reciente, úsala y cita la nueva
  3. Si no encuentras actualización, aplica el protocolo de incertidumbre
  4. Indica al usuario: "El protocolo disponible en mi base de
     conocimiento es de [año] y puede estar desactualizado."
ÁRBOL DE DECISIÓN — ANTE CUALQUIER CONSULTA CLÍNICA
1
¿Tengo fuente RAG ≤5 años sobre esto?
→ Sí: úsala como base. Cítala. Comprueba si hay algo más reciente en web.
→ No: continúa al paso 2.
2
¿Encuentro evidencia en PubMed/Web ≤5 años?
→ Sí: sintetiza, cita con PMID o URL.
→ No: continúa al paso 3.
3
¿Es conocimiento clínico consolidado sin variación reciente?
→ Sí: responde declarando explícitamente que es conocimiento establecido sin cambios recientes.
→ No: ve al paso 4.
4
No hay fuente válida disponible
→ Declara la limitación con precisión. Indica la fuente donde buscar. No completes con suposición. Fin de la respuesta.
Sección 04b — Fuente especial

PubMed: integración como herramienta viva, no RAG estático

PubMed no es indexable en un RAG estático: contiene más de 37 millones de artículos y se actualiza diariamente. Intentar crawlearlo sería inútil e ilegal bajo los términos de NCBI. La solución correcta es configurarlo como herramienta de búsqueda en tiempo real mediante la API pública NCBI E-utilities — completamente gratuita, sin límites agresivos y diseñada exactamente para esto.

🔬
NCBI E-utilities API — acceso público y gratuito
Base URL: https://eutils.ncbi.nlm.nih.gov/entrez/eutils/
Límite sin API key: 3 requests/segundo. Con API key gratuita (registro en NCBI): 10 req/s.
Registro: ncbi.nlm.nih.gov/account → My NCBI → API Key Settings.
ENDPOINTS ÚTILES
esearch — Buscar artículos
eutils/esearch.fcgi
Devuelve IDs de artículos que coinciden con la query. Filtrable por fecha, tipo de artículo, MeSH terms.
efetch — Obtener abstracts
eutils/efetch.fcgi
Dado un PMID, devuelve el abstract completo en texto plano o XML. El agente lo lee y resume.
esummary — Metadatos rápidos
eutils/esummary.fcgi
Título, autores, revista, año, DOI. Útil para mostrar resultados antes de leer el abstract completo.
PubMed Central (PMC)
eutils/efetch.fcgi?db=pmc
Para artículos open access: devuelve el texto completo, no solo el abstract. Añadir filtro pmc[filter] en esearch.
EJEMPLO DE LLAMADA — BUSCAR + LEER ABSTRACT
NCBI E-UTILITIES — FLUJO COMPLETO 
# PASO 1: Buscar artículos relevantes (últimos 3 años, pediatría)
GET https://eutils.ncbi.nlm.nih.gov/entrez/eutils/esearch.fcgi
  ?db=pubmed
  &term=bronchiolitis+treatment+children[MeSH]
  &datetype=pdat
  &reldate=1095          # últimos 3 años
  &sort=relevance
  &retmax=5             # los 5 más relevantes
  &retmode=json
  &api_key=TU_API_KEY

# Respuesta: lista de PMIDs → ["38123456", "37891234", ...]

# PASO 2: Obtener abstracts de los PMIDs
GET https://eutils.ncbi.nlm.nih.gov/entrez/eutils/efetch.fcgi
  ?db=pubmed
  &id=38123456,37891234
  &rettype=abstract
  &retmode=text
  &api_key=TU_API_KEY

# El agente recibe los abstracts en texto y los sintetiza en la respuesta

# FILTROS PEDIÁTRICOS RECOMENDADOS — añadir a cualquier query
child[MeSH]              → restringe a población pediátrica
infant[MeSH]             → lactantes
newborn[MeSH]            → neonatos
systematic review[pt]    → solo revisiones sistemáticas

free full text[filter]   → solo artículos con texto completo gratuito
ADDENDUM AL SYSTEM PROMPT — INSTRUCCIONES PARA PUBMED
SYSTEM PROMPT — MÓDULO PUBMED 
## Herramienta: pubmed_search

Cuándo usar pubmed_search:
- El usuario pregunta por evidencia científica reciente sobre un tema clínico
- Necesitas validar o actualizar un protocolo del KB con evidencia primaria
- Pregunta explícita sobre estudios, ensayos o revisiones sistemáticas
- Cualquier mención a "lo que dice la evidencia", "estudios recientes",
  "revisión Cochrane", "últimas publicaciones"

Cómo construir la query PubMed:
1. Usa términos MeSH cuando sea posible (más precisos que texto libre)
2. Añade SIEMPRE uno de: child[MeSH], infant[MeSH], newborn[MeSH]
   según el contexto pediátrico
3. Para máxima calidad: añade "systematic review[pt]" o "clinical trial[pt]"
4. Limita a los últimos 3 años salvo que el usuario pida algo histórico
5. Recupera máximo 5 artículos; lee y sintetiza los abstracts

Cómo presentar resultados PubMed:
- Resume los hallazgos principales en lenguaje adaptado al usuario
- Cita: Autor principal et al., Revista (Año). PMID: XXXXXXXX
- Si hay discrepancia entre el RAG y PubMed reciente, señálala
- No presentes los abstracts en bruto; sintetiza e interpreta
- Si el artículo no está disponible en open access, indica que
  el texto completo requiere acceso institucional

Limitación importante:
Los abstracts no siempre contienen suficiente detalle clínico.
Si el abstract es insuficiente y el artículo no es open access,
indícalo y sugiere al usuario acceder vía su institución o
contactar con la biblioteca del hospital.
⚡
Configura un caché de PubMed para queries repetidas
Las consultas frecuentes ("bronquiolitis tratamiento", "sepsis neonatal") devolverán siempre los mismos artículos top durante semanas. OpenClaw permite cachear respuestas de tools con TTL configurable. Recomendado: TTL de 7 días para queries PubMed. Reduce latencia y consumo de API. Las alertas de la AEMPS no deben cachearse nunca (TTL = 0).
Sección 05

Configuración paso a paso en OpenClaw

  1. Crear el Knowledge Base (RAG Store) En OpenClaw → Knowledge Bases → New KB. Nombra: pedienfIA-clinical. Elige embedding model: text-embedding-3-large (OpenAI) o equivalente. Chunk size recomendado: 512 tokens con 64 de overlap. Activa metadata tagging por fuente.
  2. Cargar las fuentes por prioridad Prioridad 1 primero (AEPED, SENEO, SECIP, AEMPS). Descarga PDFs de cada fuente, súbelos al KB. Para fuentes con URL pública, usa el crawler de OpenClaw con profundidad 2 y filtro por extensión .pdf. Espera confirmación de indexación antes de pasar a prioridad 2.
  3. Configurar el agente y conectar el KB Agents → New Agent → PediEnf IA. En el campo System Prompt: pega el prompt base completo (versión 1.0) seguido del Addendum de Herramientas (Sección 04). En Tools: activa Knowledge Base Search → selecciona pedienfIA-clinical.
  4. Activar Web Search En Tools del agente: activa Web Search. Si OpenClaw lo permite, configura dominios prioritarios (whitelist): aeped.es, aemps.es, secip.com, se-neonatal.es, nice.org.uk, ema.europa.eu, pubmed.ncbi.nlm.nih.gov. Esto prioriza fuentes clínicas fiables sobre resultados genéricos.
  5. Configurar parámetros del LLM Model: GPT-4o o Claude Sonnet 4 (el de mayor calidad disponible). Temperature: 0.3 (respuestas clínicas precisas, sin alucinaciones creativas). Max tokens: 1500 para respuestas estándar; 3000 si permites resúmenes de protocolos. Top-p: 0.9.
  6. Testear con preguntas de validación Antes de activar, prueba estas 5 consultas de validación (ver tabla en sección siguiente). Verifica que: (a) cita fuentes correctamente, (b) usa RAG para dosis, (c) usa web para vacunas, (d) no alucina nombres de medicamentos, (e) activa el aviso 112 ante urgencia.
  7. Programar actualización del KB Las fuentes clínicas se actualizan periódicamente. Configura un job de re-indexación trimestral para AEPED, SENEO y SECIP. Para calendario vacunal AEP: re-indexar cada enero. Para AEMPS: el web_search en tiempo real cubre las alertas activas.

Sección 07

Consideraciones críticas de seguridad clínica

⚠️
Alucinación farmacológica — riesgo máximo
Los LLMs pueden inventar dosis pediátricas con aparente confianza. El RAG conectado a Pediamécum/BNFc es la defensa principal. Si el KB no devuelve resultado para un fármaco, el agente debe indicar explícitamente que no tiene fuente verificada y derivar al profesional a Pediamécum directamente. Nunca debe responder dosis desde memoria si el KB no confirma.
⚡
Latencia en urgencias — configura timeout bajo
En preguntas de urgencia vital, cualquier llamada a herramienta añade 1-3 segundos de latencia. El sistema debe detectar patrones de urgencia (convulsión activa, shock, parada cardiorrespiratoria) y emitir el aviso de emergencias antes de cualquier búsqueda. Configura un classifier de urgencia como pre-step si OpenClaw lo permite.
🔄
Protocolo desactualizado en el KB
Un KB sin mantenimiento puede devolver guías obsoletas con más confianza que la web actualizada. La instrucción de usar web_search para confirmar vigencia de protocolos RAG es clave. Añade la fecha de indexación como metadata de cada documento y muéstrala en las citas.
✅
Disclaimer automático en respuestas clínicas
Para consultas sobre casos clínicos reales (identificables por contexto: "mi hijo", "el paciente", "tengo un niño"), el agente debe añadir automáticamente al final: "Esta información no sustituye la valoración presencial de un profesional sanitario." Esto no aplica a preguntas formativas o de protocolo general.
